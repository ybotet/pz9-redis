## 1. Описание работы

### Цель

Реализовать систему **распределённого кэширования с использованием Redis** для сервиса `tasks`, применяя паттерн **cache-aside** с TTL и jitter, обеспечивая **контролируемую деградацию** при недоступности Redis.

### Используемые технологии

| Технология     | Версия   | Назначение                  |
| -------------- | -------- | --------------------------- |
| Go             | 1.25     | Язык программирования       |
| Redis          | 7-alpine | Распределённый кэш в памяти |
| PostgreSQL     | 15       | Основная база данных        |
| Docker         | 24+      | Контейнеризация             |
| Docker Compose | 2.x      | Оркестрация сервисов        |
| gorilla/mux    | latest   | HTTP-роутер                 |
| go-redis/redis | v9       | Клиент Redis для Go         |

### Объём реализации

- Настройка Redis как сервиса в docker-compose
- Интеграция клиента Redis в сервис `tasks`
- Реализация cache-aside для `GET /v1/tasks/{id}`
- TTL с jitter для предотвращения cache avalanche
- Автоматическая инвалидация при операциях `PATCH` и `DELETE`
- Элегантная деградация при отказах Redis
- Полная документация

## 4. Реализация кэша

### 4.1. Ключи кэша

Формат:

```text
tasks:task:{id}
```

Пример:

```text
tasks:task:794ca18d-33b6-46a4-90ec-3316caf45390
```

Обоснование:

- Префикс tasks:task: позволяет легко группировать и управлять ключами
- Включение ID гарантирует уникальность и прямой доступ
- Упрощает выборочную инвалидацию

### 4.2. Паттерн cache-aside

Паттерн cache-aside (также известный как lazy loading) реализован в task_service.go:

```go
func (s *TaskService) GetTaskByID(ctx context.Context, id, userID string) (*models.Task, error) {
    // 1. Попытка чтения из Redis
    taskCache, err := s.cache.GetTask(ctx, id)
    if err == nil {
        log.Printf("[CACHE HIT] Задача %s получена из Redis", id)
        return convertToTask(taskCache), nil
    }

    //(.....)

    // 2. Cache miss: обращение к PostgreSQL
    log.Printf("[CACHE MISS] Задача %s не найдена в Redis, обращение к БД", id)
    task, err := s.repo.GetByID(id)
    if err != nil {
        return nil, err
    }

    //(.....)

    // 3. Сохранение в кэш для будущих запросов
    go func() {
        if err := s.cache.SetTask(context.Background(), task); err != nil {
            log.Printf("[CACHE WARN] Не удалось сохранить задачу %s в кэш: %v", id, err)
        }
    }()

    return task, nil
}
```

### 4.3. TTL и jitter

Конфигурация:

```go
baseTTL := 120 * time.Second
jitter := 30 * time.Second
ttl := baseTTL + time.Duration(rand.Int63n(int64(jitter)))
```

Назначение jitter:

- Предотвращает одновременное истечение всех ключей (cache avalanche)
- Распределяет нагрузку на базу данных во времени
- Повышает стабильность при высокой конкуренции

### 4.4. Инвалидация кэша

При операциях изменения выполняется инвалидация соответствующего ключа:

```go
func (s *TaskService) UpdateTask(task *models.Task) error {
// Обновление PostgreSQL
if err := s.repo.Update(task); err != nil {
return err
}

    // Инвалидация кэша
    go func() {
        if err := s.cache.DeleteTask(context.Background(), task.ID); err != nil {
            log.Printf("[CACHE WARN] Не удалось инвалидировать кэш: %v", err)
        }
    }()

    return nil

}
```

Endpoints, выполняющие инвалидацию:

- PATCH /v1/tasks/{id} → инвалидирует ключ этой задачи
- DELETE /v1/tasks/{id} → инвалидирует ключ этой задачи

### 4.5. Деградация при отказах Redis

Сервис спроектирован так, чтобы продолжать работу даже при недоступности Redis:

```go
// Инициализация
if err := redisCache.Ping(context.Background()); err != nil {
log.Printf("[WARN] Не удалось подключиться к Redis: %v. Сервис будет работать без кэша.", err)
} else {
log.Printf("[INFO] Подключено к Redis на %s", redisAddr)
}

// При операциях с кэшем
if err != nil && err.Error() != "redis: nil" {
// Ошибка Redis (не просто miss)
log.Printf("[CACHE ERROR] Redis недоступен: %v. Используется БД", err)
// Продолжаем работу с базой данных
}
```

Поведение:

- Если Redis недоступен → логируется предупреждение, используется только PostgreSQL
- Если Redis отвечает с ошибкой → логируется, кэш игнорируется
- Операции записи не завершаются ошибкой при недоступности Redis

## 6. Ответы на контрольные вопросы

1. **Что такое cache-aside и почему он используется?**

Cache-aside — это паттерн доступа к кэшу, при котором приложение самостоятельно отвечает за загрузку данных в кэш.

### Алгоритм

- Приложение сначала пытается прочитать данные из кэша.
- Если данные найдены (`cache hit`), они возвращаются.
- Если не найдены (`cache miss`), данные получаются из базы данных.
- Затем данные сохраняются в кэш для будущих запросов.

### Почему используется

- **Явный контроль:** приложение решает, что и когда кэшировать.
- **Простота:** легко реализовать и понять.
- **Эффективность:** кэшируются только реально запрашиваемые данные.
- **Консистентность:** источником истины остаётся база данных.

2. **Почему важно использовать TTL для ключей кэша?**

TTL (Time To Live) важен, потому что:

- **Предотвращает устаревание данных:** данные автоматически удаляются, обеспечивая периодическое обновление.
- **Контроль памяти:** освобождает место в Redis, удаляя неиспользуемые данные.
- **Согласованность:** позволяет обновлять данные без явной инвалидации.
- **Самовосстановление:** при ошибках инвалидации данные обновляются автоматически.

3. **Что такое cache avalanche и как помогает jitter?**

Cache avalanche — это явление, когда множество ключей кэша истекают одновременно, вызывая пиковую нагрузку на базу данных, которая может привести к её отказу.

### Как помогает jitter

- Добавляет случайный фактор к TTL каждого ключа.
- Распределяет истечение времени жизни ключей во временном диапазоне.
- Предотвращает одновременное истечение всех ключей.
- Сглаживает нагрузку на базу данных.

**Пример:** вместо того чтобы 1000 ключей истекли ровно через 120 секунд, они истекают в интервале от 120 до 150 секунд.

4. **Почему Redis не должен быть "источником истины"?**

Redis не должен быть источником истины, потому что:

- **Волатильность:** Redis хранит данные в оперативной памяти, которая теряется при перезапуске.
- **Опциональная персистентность:** сохранение на диск не гарантировано по умолчанию.
- **Слабая согласованность:** не поддерживает полные ACID-транзакции как PostgreSQL.
- **Отсутствие ссылочной целостности:** нет constraints, таких как foreign keys.
- **Назначение:** Redis оптимизирован для скорости, а не для долговременного хранения.

Вместо этого PostgreSQL (или основная база данных) должна быть источником истины, так как гарантирует долговременное хранение, согласованность и целостность.

5. **Как должен вести себя сервис при отказе Redis?**

Правильно спроектированный сервис должен:

- **Не падать:** сервис должен продолжать обрабатывать запросы.
- **Деградировать элегантно:** использовать базу данных напрямую.
- **Логировать ошибки:** записывать предупреждения для мониторинга.
- **Пытаться переподключиться:** реализовывать механизмы повторного подключения.
- **Изолировать отказы:** операции с Redis не должны блокировать запросы.
